## 4.1 为什么需要存储层级？

### 两个矛盾的需求

CPU 需要数据**越快越好**，但又需要**越多越好**——更快和更大，在物理上互相矛盾。

| | 快 | 多 |
|---|-----|-----|
| **寄存器** | ✅ 0.3ns（CPU内部） | ❌ 几十个，总共几百字节 |
| **RAM** | ❌ ~100ns | ✅ 几GB~几百GB |
| **SSD** | ❌ ~100µs | ✅ 几百GB~几TB |
| **HDD** | ❌ ~10ms | ✅ 几TB~几十TB |

**解决方案？不去解决——分级！** 把常用的放在快的里，不常用的放在大的里。

## 4.2 存储金字塔

```
                   ┌──────────────┐
         最快/最贵  │   寄存器      │  <1 ns,  ~1 KB
         / 最小    ├──────────────┤
        /          │  L1 Cache    │  ~1 ns,   ~64 KB
       /           ├──────────────┤
      /    片内    │  L2 Cache    │  ~7 ns,   ~256 KB-1 MB
     /            ├──────────────┤
    /     片外    │  L3 Cache    │  ~30 ns,  ~8-32 MB
   /             ├──────────────┤
  /              │  主存 (RAM)   │  ~100 ns, ~8-64 GB
 /               ├──────────────┤
/      最慢/最便宜 │  固态硬盘(SSD) │  ~100 µs, ~256 GB-2 TB
▼       / 最大    ├──────────────┤
                  │  机械硬盘(HDD) │  ~10 ms,  ~1-20 TB
                  ├──────────────┤
                  │  网络/云存储   │  >10 ms,  近乎无限
                  └──────────────┘
```

> 🔑 速度每降一级，大约慢 10~100 倍。但从 L1 到 HDD，差了 **1000 万倍**。

## 4.3 局部性原理（Locality）

为什么分级能工作？因为程序的行为有规律——**局部性原理**。

### 时间局部性（Temporal Locality）

> 刚刚访问过的数据，很可能很快再次被访问。

```c
for (int i = 0; i < 1000000; i++) {
    sum += arr[i];   // sum 被反复访问 —— 时间局部性
}
```

→ **策略**：把最近访问的数据保留在 Cache 里。

### 空间局部性（Spatial Locality）

> 如果一个地址被访问，它附近的地址也很可能即将被访问。

```c
for (int i = 0; i < 1000000; i++) {
    sum += arr[i];   // arr 按顺序访问 —— 空间局部性
}
```

→ **策略**：一次读一整块（Cache Line），而不是只读一个字节。

### 为什么这很重要？

```
没有局部性的程序（随机访问大数组）:
  → Cache 命中率低 → CPU 大量时间等内存 → 慢

有局部性的程序（顺序访问）:
  → Cache 命中率高 → CPU 很少等内存 → 快

同样的算法，不同数据访问顺序，性能可能差 10~100 倍！
```

## 4.4 Cache 的工作原理

### Cache Line（缓存行）

Cache 不按字节存取，而是按**块（Cache Line / Block）**，通常 64 字节。

```
内存被分成固定大小的块：
  块0 (地址 0-63), 块1 (64-127), 块2 (128-191), ...

Cache 由若干槽位(Slot)组成，每个槽位存放：
  ┌──────┬─────────┬──────────────┐
  │ V(有效)│ Tag(标签)│ Data(64字节)  │
  └──────┴─────────┴──────────────┘
```

### 映射方式

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **直接映射** | 每块内存只能放一个固定槽位 (slot = 地址 % 槽位数) | 简单、快 | 冲突多 |
| **全相联** | 任意块可以放任意槽 | 最灵活 | 查找慢（要搜所有槽） |
| **组相联** | N 个槽一组，块任意放组内（每组 N 路） | 折中方案 | — |

```
直接映射：                       4路组相联：
 内存块 → 固定槽                   内存块 → 固定组 → 组内任选
                                  
  块0 → 槽0                       块0 → 组0 → [槽0|槽1|槽2|槽3]
  块8 → 槽0 (冲突!)                块8 → 组0 → [槽0|槽1|槽2|槽3] (可共存!)
  块16→ 槽0 (冲突!)
```

### 缓存读取流程

```
CPU 发出地址 0x1000：
  1. 计算: 块号 = 0x1000 >> 6 (除以64)
  2. 计算: 目标槽/组 (取决于映射方式)
  3. 比较 Tag: 匹配? → Cache Hit! (快)
               不匹配? → Cache Miss (慢，需要从下级读)
  4. Hit: 直接返回数据
  5. Miss: 从下級存储读一整块(64B)进来，替换旧块，返回数据
```

### 替换策略

当 Cache 满时需要踢掉一个：

| 策略 | 规则 | 特点 |
|------|------|------|
| **LRU** | 踢最久没用的 | 最优但硬件成本高 |
| **FIFO** | 踢最早来的 | 简单 |
| **Random** | 随机踢 | 简单，效果不差 |
| **LFU** | 踢用得最少的 | 需要计数器 |

> 真实的 L1/L2 Cache 通常用**伪 LRU**（近似 LRU，硬件实现简单）。

## 4.5 写策略

CPU 写数据时，Cache 和内存如何保持一致？

### Write-Through（写穿透）

> 同时写入 Cache 和内存。简单，但慢（每次都要等内存）。

```
CPU 写 → Cache + 内存 (同步更新)
```

### Write-Back（写回）

> 只写 Cache，标记为"脏(Dirty)"，等被踢出时才写回内存。

```
CPU 写 → 只写 Cache，标 Dirty
Cache 替换 → 把 Dirty 块写回内存
```

### Write-Allocate vs No-Write-Allocate

| | Write-Allocate | No-Write-Allocate |
|---|---------------|-------------------|
| **写 Miss 时** | 先把块从内存读进 Cache，再写 | 直接绕开 Cache 写内存 |
| **搭配** | 常配 Write-Back | 常配 Write-Through |

## 4.6 虚拟内存

### 为什么需要虚拟内存？

- **隔离**：每个进程以为独占全部内存（不会干扰其他进程）
- **简化**：程序不用关心物理内存碎片（OS 负责映射）
- **扩展**：可以用的内存超过物理 RAM（部分放磁盘上）

### MMU（内存管理单元）

```
CPU 发出         MMU 翻译           实际访问
虚拟地址  ────→  虚拟→物理  ────→   物理地址
(VA)            页表查询            (PA)
```

### 分页（Paging）

```
虚拟地址空间被分成固定大小的「页」(通常 4KB)
物理内存被分成同样大小的「页框」(Page Frame)

虚拟地址:  ┌──────────┬──────────┐
           │ 页号(VPN) │ 页内偏移  │
           └─────┬────┴─────┬────┘
                 │          │
        页表查询  │          │ 直接复制
                 ▼          ▼
物理地址:  ┌──────────┬──────────┐
           │页框号(PFN)│ 页内偏移  │
           └──────────┴──────────┘
```

### TLB（快表）

页表存在内存中，每次内存访问查一次页表太慢了。

→ **TLB（Translation Lookaside Buffer）**：页表的专用 Cache，在 MMU 内部。只有几十到几百条，但命中率极高（>99%）。

## 4.7 实战：Cache 对性能的影响

### 矩阵乘法的访问模式

```
行优先遍历 (Cache友好):         列优先遍历 (Cache不友好):
for i in rows:                  for j in cols:
    for j in cols:                  for i in rows:
        sum += A[i][j] * ...            sum += A[i][j] * ...

A[0][0], A[0][1], A[0][2]...   A[0][0], A[1][0], A[2][0]...
连续访问 → 空间局部性好         跳跃访问 → 每次可能 Cache Miss

性能差异: 容易达到 3~10 倍！
```

```
存储金字塔（快→慢，小→大）：
  寄存器 → L1 → L2 → L3 → RAM → SSD → HDD

局部性原理：
  - 时间局部性: 刚访问的可能再访问 → 保留在 Cache
```
