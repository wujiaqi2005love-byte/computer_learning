## 5.1 为什么需要流水线？

### 没有流水线的 CPU

在不使用流水线的情况下，CPU 一次只能处理一条指令，必须等这条指令完全结束后才能开始下一条：

```
指令1: [===取指===][===译码===][===执行===][===写回===]
指令2:                                                [===取指===][===译码===][===执行===][===写回===]
指令3:                                                                                             ...

时间 ─────────────────────────────────────────────────────────────────────────────→
```

结果：大量硬件在"围观"——当指令在"执行"时，取指和译码的硬件闲置着。

### 有流水线的 CPU

```
指令1: [===取指===][===译码===][===执行===][===写回===]
指令2:            [===取指===][===译码===][===执行===][===写回===]
指令3:                        [===取指===][===译码===][===执行===][===写回===]
指令4:                                    [===取指===][===译码===][===执行===][===写回===]

时间 ─────────────────────────────────────────────────────────────────────────────→
```

> 🔑 四倍吞吐量！每个时钟周期都能完成一条指令（理想情况）。

就像洗衣服：洗衣机在转的时候，你可以同时烘干上一批。

## 5.2 经典五级流水线（MIPS 风格）

```
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│    IF    │    ID    │    EX    │    MEM   │    WB    │
│Instruction│Instruction│ Execute  │ Memory  │ Write    │
│  Fetch   │  Decode  │          │ Access  │  Back    │
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ 取指令    │ 译码+读   │ ALU运算   │ 读/写内存 │ 写回寄存器 │
│ PC→IMEM  │ 寄存器    │          │          │          │
└──────────┴──────────┴──────────┴──────────┴──────────┘
```

| 阶段 | 缩写 | 做了什么 | 关键硬件 |
|------|------|----------|----------|
| **取指** | IF | 从内存取指令，PC+1 | 指令 Cache、PC 加法器 |
| **译码** | ID | 解析指令，读寄存器 | 寄存器文件、控制器 |
| **执行** | EX | ALU 运算，计算地址 | ALU |
| **访存** | MEM | LOAD/STORE 访问数据 Cache | 数据 Cache |
| **写回** | WB | 结果写入目标寄存器 | 寄存器文件写端口 |

## 5.3 流水线冒险（Pipeline Hazards）

流水线不是免费的，有三类"冒险"会导致流水线停顿（stall）。

### 结构冒险（Structural Hazard）

> 两条指令争抢同一个硬件资源。

```
指令1: LOAD R0, [addr]   ← 需要访存 (MEM 阶段用数据 Cache)
指令2: (下一条要取指)     ← 也需要访存 (IF 阶段用指令 Cache)

解决: 哈佛架构 → 分开指令 Cache 和数据 Cache
```

### 数据冒险（Data Hazard）

> 后面的指令需要前面指令的计算结果，但结果还没产生。

```
ADD  R1, R2, R3    ; R1 在 WB 阶段才写入
SUB  R4, R1, R5    ; 但 ID 阶段就需要读 R1（ADD 还没写完！）

时间线：  ADD:  IF  ID  EX  MEM  WB
          SUB:      IF  ID  EX  MEM  WB
                        ↑
                     需要 R1，但 ADD 的 WB 还没到！
```

**解决方法：**
| 方法 | 原理 | 代价 |
|------|------|------|
| **停顿 (Stall/Bubble)** | 插入 NOP，等结果出来 | 性能损失 |
| **转发 (Forwarding/Bypassing)** | ALU 结果直接传给下一条指令的 EX | 额外数据通路 |
| **编译器调度** | 编译器重排指令，把无关指令插入危险指令之间 | 编译器更复杂 |

```
转发的工作方式：
  ADD  R1, R2, R3    在 EX 阶段就得到结果
  SUB  R4, R1, R5    EX 阶段需要的 R1 直接从 ADD 的 EX/MEM 流水线寄存器"偷"过来
                     不需要等到 WB！
```

### 控制冒险（Control Hazard）

> 不知道下一条指令是什么，因为跳转结果还没出来。

```
    CMP  R1, R2       ; 条件判断
    BEQ  label        ; 跳不跳？EX 阶段才知道
    ADD  R3, R4, R5   ; ← 如果跳转，这条不该执行！

时间线： BEQ: IF  ID  EX  MEM  WB  ← 跳转方向在 EX 才知道
         下一条:    IF  ID  ...     ← 但这时候已经取了指令！
```

**解决方法：**
| 方法 | 原理 |
|------|------|
| **冲刷 (Flush)** | 猜错了就把流水线中那些不该执行的指令清掉 |
| **分支预测** | 猜跳转方向，猜对的概率高达 90%+ |
| **延迟分支** | 跳转指令后面的那条指令无论如何都会执行（编译器填上一条有用的） |

## 5.4 分支预测

现代 CPU 的分支预测器极其复杂，准确率 > 95%。

### 简单策略

| 预测器 | 原理 | 准确率 |
|--------|------|--------|
| **静态-总不跳** | 假设分支总是不跳 | ~40% |
| **静态-向后跳** | 循环向回跳 → 跳转，if判断前跳 → 不跳 | ~65% |
| **1-bit 预测** | 上次跳了这次就跳 | ~80% |
| **2-bit 预测** | 连续两次预测错才改变方向 | ~90% |
| **全局/混合预测** | 结合分支历史 + 全局模式 | >95% |

```
2-bit 饱和计数器状态机：

    预测不跳 ←── 实际跳转 ──→ 预测不跳 ←── 实际跳转 ──→ 预测跳转
     (00)          →          (01)          →          (10)
       ↑                                                   │
       └───────────────────────────────────────────────────┘
                    实际跳转（从10→11: 预测跳转）
       ┌───────────────────────────────────────────────────┐
       │                                                   ↓
    预测不跳 ←── 实际不跳 ──→ 预测不跳 ←── 实际不跳 ──→ 预测跳转
     (00)          ←          (01)          ←          (10)
```

## 5.5 超标量（Superscalar）

> 一个时钟周期内发射多条指令。硬件有多条流水线。

```
普通标量（1条/周期）：         超标量（2条/周期）：
 ┌─┐ ┌─┐ ┌─┐                   ┌─┐┌─┐ ┌─┐┌─┐ ┌─┐┌─┐
 │A│ │B│ │C│                   │A││B│ │C││D│ │E││F│
 └─┘ └─┘ └─┘                   └─┘└─┘ └─┘└─┘ └─┘└─┘
 时间 →                         时间 →
```

| 特性 | 标量 | 超标量 |
|------|------|--------|
| 发射宽度 | 1 指令/周期 | 2~8 指令/周期 |
| 硬件 | 1 条流水线 | 多条并行流水线 |
| 挑战 | — | 数据依赖、资源冲突 |
| 实例 | 早期 ARM | Intel Core i9, Apple M 系列 |

### 乱序执行（Out-of-Order Execution）

超标量的自然延伸：允许后面不依赖前面结果的指令先执行。

```
程序顺序:                    实际执行顺序:
1. LOAD R1, [慢内存]         1. LOAD R1, [慢内存]  ← 派出去等着
2. ADD R2, R3, R4             3. SUB R5, R6, R7    ← 不依赖R1，先执行!
3. SUB R5, R6, R7             2. ADD R2, R3, R4    ← 也不依赖R1，执行!
4. MUL R8, R1, R5             4. MUL R8, R1, R5    ← R1到了，执行
```

> **核心要求**：最终结果要和顺序执行一致（精确中断）。

## 5.6 SIMD — 数据级并行

> 一条指令同时操作多个数据。

```
标量操作 (SISD):
  A[0] + B[0] = C[0]
  A[1] + B[1] = C[1]
  A[2] + B[2] = C[2]
  A[3] + B[3] = C[3]    ← 4 条指令

SIMD 操作 (一条指令):
  [A0 A1 A2 A3] + [B0 B1 B2 B3] = [C0 C1 C2 C3]  ← 1 条指令！
```

| 指令集 | 宽度 | 简称 |
|--------|------|------|
| x86 SSE | 128 bit | 一次处理 4 个 float |
| x86 AVX/AVX2 | 256 bit | 一次处理 8 个 float |
| x86 AVX-512 | 512 bit | 一次处理 16 个 float |
| ARM NEON | 128 bit | 一次处理 4 个 float |
| RISC-V V | 可扩展 | 向量长度可配 |

## 5.7 多核与超线程

### 多核（Multi-core）

> 一个芯片上放多个独立的 CPU 核心。每个核心有自己的 L1/L2 Cache，共享 L3。

```
┌────────────────── Chip ────────────────────┐
│                                            │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐       │
│  │Core 0│ │Core 1│ │Core 2│ │Core 3│       │
│  │L1 L2 │ │L1 L2 │ │L1 L2 │ │L1 L2 │       │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘       │
│     └────────┴────────┴────────┘            │
│                  │                          │
│          ┌───────▼────────┐                 │
│          │  L3 Cache (共享) │                │
│          └───────┬────────┘                 │
│                  │                          │
│          ┌───────▼────────┐                 │
│          │  内存控制器 + RAM │                │
│          └────────────────┘                 │
└────────────────────────────────────────────┘
```

### 超线程（SMT / Hyper-Threading）

> 一个物理核心伪装成两个逻辑核心。当一个线程在等内存时，另一个线程用 ALU。

```
物理核心:
  ┌───────────────┐
  │ ALU ← 线程A    │  ← 线程A 的计算在用 ALU
  │ 取指 ← 线程B    │  ← 同时线程B 在用取指单元（线程A 在等内存）
  │ 译码 ← 线程B    │
  │ 访存 ← 线程A    │  ← 线程A 的数据在内存路上
  └───────────────┘

效果：单核 15~30% 性能提升（几乎免费！），但两个线程争一个 Cache。
```

## 5.8 Flynn 分类法

| 类别 | 含义 | 例子 |
|------|------|------|
| **SISD** | 单指令单数据 | 早期 CPU |
| **SIMD** | 单指令多数据 | GPU 核心、AVX/NEON |
| **MISD** | 多指令单数据 | 容错系统（罕见） |
| **MIMD** | 多指令多数据 | 多核 CPU、分布式系统 |

```
SISD:  1 PE 做 1 件事          SIMD:  1 条指令控制多个 PE
       ┌─┐                        ┌─┐┌─┐┌─┐┌─┐
       │A│                        │A││A││A││A│  ← 同一指令
       └─┘                        └─┘└─┘└─┘└─┘
      数据A                        ↓  ↓  ↓  ↓
                                  数据 数据 数据 数据

MIMD:  多 PE 各做各              现代 CPU = MIMD + 每核内 SIMD
       ┌─┐┌─┐┌─┐┌─┐
       │A││B││C││D│  ← 不同指令
       └─┘└─┘└─┘└─┘
       ↓  ↓  ↓  ↓
      各处理各自数据
```

## 📝 本章速查

```
流水线: IF → ID → EX → MEM → WB（重叠执行，提高吞吐量）

三种流水线冒险：
  - 结构冒险: 硬件资源冲突 → 哈佛架构
  - 数据冒险: 前一条结果未就绪 → 转发(Forwarding) / 停顿
  - 控制冒险: 跳转方向未知 → 分支预测 / 延迟分支

分支预测: 2-bit 饱和计数器，现代预测器 >95%准确率

超标量: 每周期发射多条指令 → 乱序执行

SIMD: 一条指令操作多个数据 (AVX, NEON)

多核: 多独立核心，共享 L3 + 内存
超线程(SMT): 1物理核 → 2逻辑核，~15-30% 免费提升

并行层次: 指令级(流水线) → 数据级(SIMD) → 线程级(超线程) → 核心级(多核)
```
